snabbdom 简介

  • snabbdom 是瑞典语单词，单词原意 “速度”。

  • snabbdom 是著名的虚拟 DOM 库，是 diff 算法的鼻祖，Vue 源码借鉴了 snabbdom。

  • 官方 github：https://github.com/snabbdom/snabbdom。

  • 在 github 上的 snabbdom 源码是用 TypeScript 写的，github 上并不提供编译好的 JavaScript 版本。

  • 如果要直接使用 build 出来的 JavaScript 版的 snabbdom 库，可以从 npm 上下载：

npm i -D snabbdom

搭建项目环境

snabbdom 库是 DOM 库，当然不能在 nodejs 环境运行，所以我们需要搭建 webpack 开发环境，好消息是不需要安装任何 loader。

跑通 snabbdom 官方 demo

import {
  init,
  classModule,
  propsModule,
  styleModule,
  eventListenersModule,
  h
} from "snabbdom";

const patch = init([
  // Init patch function with chosen modules
  classModule, // makes it easy to toggle classes
  propsModule, // for setting properties on DOM elements
  styleModule, // handles styling on elements with support for animations
  eventListenersModule // attaches event listeners
]);

const container = document.getElementById("container");

const vnode = h(
  "div#container.two.classes",
  { on: { click: () => console.log("div clicked") } },
  [
    h("span", { style: { fontWeight: "bold" } }, "This is bold"),
    " and this is just normal text",
    h("a", { props: { href: "/foo" } }, "I'll take you places!")
  ]
);
// Patch into empty DOM element – this modifies the DOM as a side effect
patch(container, vnode);

const newVnode = h(
  "div#container.two.classes",
  { on: { click: () => console.log("updated div clicked") } },
  [
    h(
      "span",
      { style: { fontWeight: "normal", fontStyle: "italic" } },
      "This is now italic type"
    ),
    " and this is still just normal text",
    h("a", { props: { href: "/bar" } }, "I'll take you places!")
  ]
);
// Second `patch` invocation
patch(vnode, newVnode); // Snabbdom efficiently updates the old view to the new state

虚拟 Dom 和 h 函数

虚拟 Dom

什么是 虚拟 Dom？如下图所示：

虚拟 Dom 就是用 JavaScript 对象描述 Dom 的层次结构。Dom 中的一切属性都在 虚拟 Dom 中有对应的属性。

为什么要有 虚拟 Dom ？

其实原因就是一句话：diff 是发生在 虚拟 Dom 上的。也就是节点和节点进行 精细化比较（diff） 并不是 真实 Dom 之间的比较。

而是：新虚拟 DOM 和 旧虚拟 DOM 进行 diff（精细化比较），算出应该如何 最小量更新，最后反映到 **真实 DOM**上。如下图所示：

注意：这里不研究 Dom 是如何转变成为 虚拟 Dom 的，因为这个属于 模板编译 原理范畴，本文档不研究。

这里我们主要研究以下几点：

  1. 虚拟DOM 如何被 **渲染函数（h 函数）**产生？
    • 我们要手写 h 函数
  2. diff 算法 原理？
    • 我们要手写 diff 算法
  3. 虚拟 DOM 如何通过 diff 变为真正的 DOM 的。
    • 事实上，虚拟 DOM 变回真正的 DOM，是涵盖在 diff 算法 里面的。

h 函数

h 函数 是用来产生 **虚拟节点（vnode）**的。

比如，我们可以这样调用 h 函数：

import { h } from "snabbdom";

h('a', { props: { href: 'http://www.baidu.com' }}, '百度');

上面的 a、{ props: { href: 'http://www.baidu.com' }}、百度 分别对应 h 函数 的三个参数。

  • 第一个参数：表示的要最终生成 真实 Dom 的标签的名称。
  • 第二个参数：是一个对象，对象里边还有一个 props 对象，props 对象表示该标签的有哪些属性。
  • 第三个参数：表示该标签中插入的内容。

它将会得到以下的 虚拟节点：

{
  sel: "a",
  elm: undefined,
  data: {
	props: {
	  href: 'http://www.baidu.com'
	}
  },
  key: undefined,
  children: undefined,
  text: "百度"
}

而这个 虚拟节点 表示的真正的 Dom 节点如下：

<a href="http://www.baidu.com">百度</a>

注意：目前只通过 h 函数 创建的虚拟节点并不会直接显示在页面上。 后续我们会通过创建一个 patch 函数通过该虚拟节点渲染真正的dom 节点。

注意：第二个参数没有内容直接省略不传，如下所示：

// 创建一个虚拟节点
const vnode = h(
  'div', // 第一个参数
  '我是一个div' // 这是第三个参数，第二个参数省略了
)

虚拟节点属性说明

  • children：表示该虚拟节点的子元素，undefined 表示没有子元素。
  • data：表示虚拟节点的属性和样式的配置。
  • elm：表示虚拟节点对应的真正节点。如果为 undefined 表示这个虚拟节点还没有上树。
  • key：表示当前虚拟节点的 唯一标识。
  • sel：表示选择器。
  • text：表示标签内容文字。

渲染虚拟节点

之前说了 h 函数 仅仅是帮助我们创建了一个 虚拟节点，要想把这个 虚拟节点 显示在页面上，还需要我们创建一个 patch 函数，通过这个函数来渲染 虚拟节点 对应的 真实节点：

// /src/index.js

import {
  init,
  classModule,
  propsModule,
  styleModule,
  eventListenersModule,
  h
} from "snabbdom";

// 创建一个虚拟节点
const vnode = h(
  'a',
  {
    props: {
      href: 'http://www.baidu.com'
    }
  },
  '百度'
)

// 创建一个 patch 函数
const patch = init([
  classModule,
  propsModule,
  styleModule,
  eventListenersModule,
])

// 之前 index.html 文件中声明 id 为 container 的 div 盒子
const container = document.querySelector('#container')

// patch 函数让虚拟节点上树
patch(container, vnode)

h 函数嵌套使用得到虚拟 DOM 树

比如我们可以这样嵌套使用 h 函数：

h(
  'ul',
  {},
  [
    h('li', {}, '牛奶'),
    h('li', {}, '咖啡'),
    h('li', {}, '可乐')
  ]
);

上述示例代码中，我们可以看到这个 h 函数 是要生成一个 ul 标签的虚拟节点，该虚拟节点第二个参数为空对象，表示不做任何属性配置，第三个参数传递的是一个数据列表，而不是我们之前传的字符串了，表示的是该虚拟节点下的子虚拟节点。

该 h 函数 将得到以下的 虚拟 dom 树：

{
  "sel": "ul",
  "data": {},
  "children": [
    { "sel": "li", "text": "牛奶" },
	{ "sel": "li", "text": "咖啡" },
	{ "sel": "li", "text": "可乐" }
  ]
}

注意如果 虚拟 Dom 树 的子元素只有一个可以不用数组的写法：

h(

  {},
  [
    h('li', {}, '牛奶'),
    h('li', {}, '咖啡'),

      h('div', [ // 有多个子元素，需要包裹在数组里

          h('p', '我是p2')

    h('li', {}, h('p', '我是p3')) // 只有一个子元素可以不用在数组里写 h 函数，当然如果写在数组里也没问题的
  ]

我们再继续看 h 函数 的源码，发现前面会有很多的 if 语句，这是在做什么呢？

其实这是在做 函数的重载，什么是 函数的重载 呢？它是 C/C++ 和 Java 等编程语言中的概念。

指的是一个函数如果有多种用法，那么我们就说这个函数是具有重载的。

函数重载 是允许在同一作用域中声明几个功能类似的同名函数，这些函数的 参数列表必须不同，常用来处理实现功能类似数据类型不同的问题。

正如我们之前代码示例中所示，h 函数 可以用多种用法，每种用法所传入的参数以及参数的个数都是不同的。

手写简易版 h 函数，没有做 函数重载 操作，所以必须要传入三个参数，只不过针对第三个参数的不同，调用的方式可以不同：

// /src/scripts/h.js

import vnode from './vnode'

/**
 * 手写 h 函数
 * @param { String} sel 选择器名称
 * @param { Object } data 节点属性和样式
 * @param { * } c 以下三种调用方式中的参数类型
 * 
 * - 调用方式有以下三种：
 * -- 1. h('div', {}, '文字')
 * -- 2. h('div', {}, [])
 * -- 3. h('div', {}, h())
 */
export default function(sel, data, c) {
  if (arguments.length < 3) throw new Error('参数有误')

  // 以下判断第三个参数的调用方式
  //
  // 调用方式 1
  if (typeof c === 'string' || typeof c === 'number') {
    return vnode(sel, data, undefined, c, undefined)
  }

  // 调用方式 2
  // 列表的每一项都是在调用 h()，所以每一项都是一个 vnode 对象
  else if (Array.isArray(c)) {
    let children = []

    for (let i = 0; i < c.length; i++) {
      // 检查 c[i] 必须是一个对象，因为 h 执行的结果就是一个 vnode 对象
      // 如果有一个不满足，报错
      if (!(typeof c[i] === 'object' && c[i].hasOwnProperty('sel'))) {
        throw new Error('第三个参数有误')
      }

      // 收集 children 项
      children.push(c[i])
    }

    return vnode(sel, data, children, undefined, undefined);
  }

  // 调用方式 3
  // 第三个参数调用 h()，所以返回的是一个 vnode 对象
  else if (typeof c === 'object' && c.hasOwnProperty('sel')) {
    const children = [c]

    return vnode(sel, data, children, undefined, undefined);
  }

  else throw new Error('第三个参数有误')
}

初步感受 Diff 算法

我们先来感受以下 Diff 算法，我们先回到 src/index.js 文件中，用回 snabbdom 提供的 h 函数 写一个 虚拟 Dom 树，然后来感受以下 Diff 算法：


import {
  init,
  h,
  classModule,
  propsModule,
  styleModule,
  eventListenersModule
} from "snabbdom";


const container = document.querySelector('#container')
const btn = document.querySelector('button')

// 创建一个 patch 函数
const patch = init([
  classModule,
  propsModule,
  styleModule,
  eventListenersModule,
])

// 创建虚拟 Dom 树1
const vnode1 = h('ul', {}, [
  h('li', {}, 'A'),
  h('li', {}, 'B'),
  h('li', {}, 'C'),
  h('li', {}, 'D'),
])

// 创建虚拟 Dom 树2
const vnode2 = h('ul', {}, [
  h('li', {}, 'A'),
  h('li', {}, 'B'),
  h('li', {}, 'C'),
  h('li', {}, 'D'),
  h('li', {}, 'E'),
])

// patch 函数让虚拟节点上树
patch(container, vnode1)

// 点击按钮，讲 vnode1 变成 vnode2

  patch(vnode1, vnode2)

那么为什么会造成这种现象呢？明明节点都没有发生变化，只是在前面加了 E，为什么会整个 Dom 会被替换呢？

原因是：因为我们没有给 虚拟 Dom 添加 key 唯一标识。

关于 Diff 和 唯一标识 key

  • key 很重要。key 是这个节点的 唯一标识，告诉 diff 算法，在更改前后它们是同一个DOM 节点。
  • 只有是 同一个 虚拟节点，才进行 精细化比较，否则就是暴力插入新的、删除旧的。
    • 延伸问题：如何定义是同一个虚拟节点？
    • 答：选择器 相同且 key 相同，例如代码中的 vnode1 和 vnode2 都是 ul 选择器元素，而我们都没有给 ul 指定 key（正常情况是要指定的），所以 key 都为 undefined，它们是同一个 虚拟节点。
    • 为什么要先插入后删除？
    • 因为：为插入的时候需要一个基准节点（也就是源节点），插入到原来节点的前面。然后再把原来的删掉。如果先删除的话就没有基准节点了。
  • 只进行 同层比较，不会进行 跨层 比较。即使是同一片虚拟节点，但是跨层了，精细化比较不 diff 你，而是暴力删除旧的、然后插入新的。

  • 新后 和 旧后：指的是 新虚拟节点 和 旧虚拟节点 中，所有没有处理的最后一个节点。
  • 所谓的没有处理指的是没有进行过精细化比较的。

查找方式说明：

  • 第一种：
    • 当 新前 和 旧前 命中后，在处理完对应的节点后，新前 和 旧前 的索引需要往下移动。
  • 第二种：
    • 当 新后 和 旧后 命中后，在处理完对应的节点后，新后 和 旧后 的索引需要往上移动。
  • 第三种：
    • 当 新后 和 旧前 命中后，在处理完对应的节点后，新后 的索引要往上移动，旧前 的索引需要往下移动。
    • 同时，新后 指向的节点，需要移动到 旧后 的后面。
  • 第四种：
    • 当 新前 和 旧后 命中后，在处理完对应的节点后，新前 的索引要往下移动，旧后 的索引需要往上移动。
    • 同时，新前 指向的节点，需要移动到 旧前 的前面。

首先，会用 旧前 和 新前 这两个节点进行对比，对比 旧前 和 新前 是不是同一个节点。

import createElements from "./createElements"
import updateChildren from "./updateChildren"

/**
 * 对 新旧相同的两个虚拟节点 进行比较
 * @param { * } oldVnode 旧的虚拟节点
 * @param { * } newVnode 新的虚拟节点
 */
const patchVnode = (oldVnode, newVnode) => {
  // 判断新旧虚拟节点是否完全一样
  // 如果一样，什么都不做，直接 return
  if (oldVnode === newVnode) return

  // 判断新虚拟节点有没有 text 属性
  // 如果新节点有 text 属性
  if (newVnode.text !== undefined && (!newVnode.children || newVnode.children.length === 0)) {
    // 如果新虚拟节点的 text 不等于 旧虚拟节点的 text
    if (newVnode.text !== oldVnode.text) {
      // 把新虚拟节点的 text 直接覆盖 旧虚拟节点的 text
      // 即使 旧虚拟节点 的 elm 中是 children，那么也会被覆盖成 text
      oldVnode.elm.innerText = newVnode.text
    }
  }
  // 如果，没有 text 属性，那么就意味着 新节点有 children
  // 因为我们写的 h 函数，必须要接受三个参数
  else {
    // 判断 旧虚拟节点 有没有 children
    // 如果 就虚拟节点 没有 children，也就是说 旧虚拟节点 有 text
    if (!oldVnode.elm.children || oldVnode.elm.children.length === 0) {

      // 清空旧节点的内容
      oldVnode.elm.innerHTML = ''

      for (let i = 0; i < newVnode.children.length; i++) {
        const child = newVnode.children[i]
        const childDom = createElements(child)
        oldVnode.elm.appendChild(childDom)
      }
    }

    // 如果 旧节点有 children，那么就是最复杂的情况
    // 这时候 新旧虚拟节点 都有 children
    else {
      // 那么这里要给 updateChildren() 方法传入的父节点是谁？
      // 因为 oldVnode 和 newVnode 是同一个节点才会进入该 patchVnode 方法，
      // 如果不管是 oldVnode 还是 newVnode 其实都是它们子元素的父节点，
      // 并且也是对方子元素的父节点。所以，这里传入 oldVode.elm 和 newVnode.elm 都行。
      updateChildren(oldVnode.elm, oldVnode.children, newVnode.children)
    }
  }
}
