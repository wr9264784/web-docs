Token 的无感刷新

如何实现 token 的无感刷新，当我们要去聊这个话题的时候，一定要知道身处一个什么样的环境中，要聊这件事，一定是在 单点登录 的环境中，脱离了 单点登录 聊 token 的无感刷新没有意义。

现在我们不考虑什么叫 单点登录，也不去考虑 token 无感刷新的意义在哪。这里我们单单就聊什么是 token 的无感刷新以及如何实现它。

这件事其实主要工作在后端，跟前端的关系不大，前端在代码层面要不要做一些处理呢？肯定是要的，只是工作量其实很小。

手动刷新 Token（非无感刷新）

以下做了一个极度简化的例子：

  1. 当点击了登录后，这里也不需要去输入什么账号密码了，接口返回一个 token，我们需要把 token 保存起来。后续点击请求受保护的接口时，就可以把这个 token 携带上，放在请求头 headers 中的 token 字段中。

  2. 这里后端接口设置了 token 的过期时间为 3s，当 3s 后再次点击请求受保护的接口，即使携带了 token 也会返回 401 错误。设置 3s 过期主要是为了验证实现 token 的无感刷新。

  3. 在实际开发中，当我们 token 过期了，要想请求受保护的接口怎么办呢？只能让用户重新登录获取新的 token 了，但是如果 token 的过期时间较短，每次都让用户重新登录体验会非常不好。所以，要使用 token 的无感刷新。

  4. 要是实现 token 的无感刷新服务器需要返回两个 token，一个是过期时间较短的 token（这个叫做 用户 token），一个是过期时间较长的 token（一般是 7 天，两周或者一个月，这个叫做 刷新 token）。刷新 token 要做的就一件事，就是当 用户 token 过期后，可以使用这个 刷新 token 重新去换一个新的 用户 token。所以登录的时候应该返回两个 token：

  5. 当我们点击 刷新 token 按钮后，就需要拿着 刷新 token 去请求，获取新的 用户 token，然后保存这个 token 替换之前过期的 token，再次请求受保护的接口就能正常返回了。

代码如下：

<template>
  <div class="refresh-token">
    <el-button type="primary" @click="onLogin">登录</el-button>
    <el-button type="warning" @click="onVerifyToken">请求受保护的接口</el-button>
    <el-button @click="onRefreshToken">刷新 token</el-button>
  </div>
</template>

<script setup lang="ts">
import { ref } from 'vue'
import axios from 'axios'

defineOptions({
  name: 'RefreshTokenView'
})

const baseUrl = 'http://localhost:7001'
const token = ref('')
const refreshToken = ref('')

const onLogin = async () => {
  const { data } = await axios.post(`${baseUrl}/api/login`)
  const { token: _token ,refreshToken: _refreshToken } = data.data
  console.log(token, refreshToken);
  
  token.value = _token
  refreshToken.value = _refreshToken
}

const onVerifyToken = async () => {
  await axios.get(`${baseUrl}/api/verifyToken`, {
    headers: {
      token: token.value
    }
  })
}

const onRefreshToken = async () => {
  const { data } = await axios.get(`${baseUrl}/api/refreshToken`, {
    headers: {
      token: refreshToken.value
    }
  })
  token.value = data.data.newToken
}
</script>

自动刷新 Token（无感刷新）

要想无感刷新 token，其实就是省略了我们手动点击 刷新 token 的按钮，当我们直接点击 请求受保护的接口 按钮时，无法发现返回的是 401，那么就自动帮我们携带 refreshToken 去请求新的 用户 token，然后再自动帮助我们请求一遍之前失败的接口。

为了可以在请求时方便自动刷新 token 并重新请求，我们可以对 axios 进行二次封装，在 src/utils/request.ts 中进行封装：

import axios from 'axios'
import { setToken, getToken, getRefreshToken, setRefreshToken } from '@/utils/index'
import { refreshTokenAPI } from '@/api/user' // 刷新 token 的请求 api
import type { AxiosResponse } from 'axios'

const axiosInstance = axios.create({
  baseURL: 'http://localhost:7001',
  timeout: 5000
})

// 请求拦截器
axiosInstance.interceptors.request.use(config => {
  const token = getToken()
  const refreshToken = getRefreshToken()
  const { isRefreshToken } = config.headers
  
  // 为了判断当前请求是否为刷新 token 的请求，我们这里可以通过判断请求的 url 是否为 '/api/refreshToken'
  // 但是这样做会造成代码之间耦合，所以这里我们通过在请求配置中的请求头添加一个 isRefreshToken 属性，来区分刷新 token 的请求
  // 如果请求的是非刷新 token 的接口，有 token 就把 token 带上
  if (token && !isRefreshToken) config.headers.token = token

  // 如果请求的是刷新 token 的接口，有 refreshToken 就把 refreshToken 带上
  else if (refreshToken && isRefreshToken) config.headers.token = refreshToken

  return config;
})

// 响应拦截器
axiosInstance.interceptors.response.use(
  async (res: AxiosResponse) => {
    const { code, data } = res.data
    
    const { isRefreshToken } = res.config.headers
    
    // 这里拦截响应，一旦拿到 token 后，保存 token
    // 如果有 用户 token
    if (data && (data.token || data.newToken)) {      
      const token = data.token || data.newToken
      setToken(token)
    }

    // 如果有 刷新 token
    if (data && data.refreshToken) {
      const refreshToken = data.refreshToken
      setRefreshToken(refreshToken)
    }

    // 如果没有权限，通过 refreshToken 刷新 token，并且重新请求接口
    if (code + '' === '401') {
      // 调用接口重新刷新 token
      await refreshTokenAPI()
      
      // 重新请之前 401 的接口
      // 把之前的请求配置 res.config 重新传入
      const resp = await axiosInstance.request(res.config)

      return resp.data
    }

    return res.data
  }
)

export default axiosInstance

在 src/utils/token.ts 中封装 token 相关的公用方法，并导出：

export const getToken = () => {
  return localStorage.getItem("token") || ""
}

export const getRefreshToken = () => {
  return localStorage.getItem("refreshToken") || ""
}

export const setToken = (token: string) => {
  localStorage.setItem("token", token)
}

export const setRefreshToken = (refreshToken: string) => {
  localStorage.setItem("refreshToken", refreshToken)
}

src/utils/index.ts 中统一导出，方便使用：

import request from './request'
import { getToken, setToken, setRefreshToken, getRefreshToken } from './token'

export {
  request,
  getToken,
  setToken,
  setRefreshToken,
  getRefreshToken
}

在 src/api/user.ts 中封装重新请求 token 的 api：

import { request } from '@/utils/index'

export const refreshTokenAPI = () => {
  return request({
    url: '/api/refreshToken',
    method: 'get',
    headers: {
      isRefreshToken: 1 // 1 表示是 刷新 token 的接口，0 或没有表示不是刷新 token 的接口
    }
  })
}

点击登录，过了 3s 后点击请求受保护的接口按钮，request.ts 的响应拦截器中发现请求结果为 401 ，自动帮助我们刷新 token，然后重新请求之前 401 的接口：

好了，现在功能大体功能已经有了，但是还有没有其它问题呢？我们来考虑一下这种情况：如果刷新 token 也过期了，怎么办呢？

我们先来把本地 localStorage 中的 refreshToken 删除掉，然后点击请求受保护的接口，发现出现死循环地请求 刷新 token：

造成这种现象的原因是因为在 src/utils/request.ts 的响应拦截器中，对于 code 为 401 的判断，因为 refreshToken 过期了，那么请求 刷新 token 的 api 返回的 code 肯定也是 401，那么就造成了死循环：

// ...省略其它

// 响应拦截器

  async (res: AxiosResponse) => {
    const { code, data } = res.data
    
    // ...省略其它

    // 如果没有权限，通过 refreshToken 刷新 token，并且重新请求接口
    if (code + '' === '401') {
      console.log('刷新 token');

      // 调用接口重新刷新 token
      await refreshTokenAPI()
      
      console.log('重新请求');
      
      // 重新请之前 401 的接口
      // 把之前的请求配置 res.config 重新传入
      const resp = await axiosInstance.request(res.config)

      return resp.data
    }

    return res.data
  }
)

所以在判断 code 为 401 的时候，还要针对不是刷新 token 的接口：

 // ...省略其它

// 响应拦截器
axiosInstance.interceptors.response.use(

我们再来把本地 localStorage 中的 refreshToken 删除掉，然后点击请求受保护的接口，发现还是出现死循环地请求，只不过这次不是只重复请求 刷新 token 的 api，而是 刷新 token 的 api 和 之前 401 的接口 一起死循环：

这又是为什么呢？我们现在没有 刷新 token，请求一个受保护的接口，然后返回 401，进入响应拦截器 401 的判断逻辑，然后去 刷新 token，问题是这个刷新 token 的接口能刷新到吗？其实是刷新不到的！

那么我都拿不到刷新后的 token 了，还有必要重新请求之前的 401 接口吗？那肯定是不能请求的！那就得走登录页面的逻辑了，让用户重新登陆。

所以，这里响应拦截器还需要做进一步的判断：

 // ...省略其它

// 响应拦截器
axiosInstance.interceptors.response.use(
  async (res: AxiosResponse) => {
    const { code, data } = res.data
    
    // ...省略其它

    const { isRefreshToken } = res.config.headers
    
    // 如果没有权限，通过 refreshToken 刷新 token，并且重新请求接口
    // 判断还要针对不是刷新 token 的请求接口
    // 如果没有权限，通过 refreshToken 刷新 token，并且重新请求接口
    if (code + '' === '401' && !isRefreshToken) {
      console.log('刷新 token');

      // 调用接口重新刷新 token
      const { data } = await refreshTokenAPI()
      
      console.log('重新请求');

      // 如果拿到了刷新后的 token
      if (data && data.newToken) {
        // 重新请之前 401 的接口
        // 把之前的请求配置 res.config 重新传入
        const resp = await axiosInstance.request(res.config)

        return resp.data
      }
      
      else {
        alert('登录过期，请重新登录')
        return res.data
      }
    }

    return res.data
  }
)

export default axiosInstance
