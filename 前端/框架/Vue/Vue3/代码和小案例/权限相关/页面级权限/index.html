页面路由权限

两种实现方式

我们在 vue 项目的后台管理系统中，一定会遇到页面路由权限的问题，页面路由权限一般可以有两种做法：

  1. 前端提前声明动态路由列表并配置对应的路由权限码，然后用户跳转路由页面的时候通过接口获取用户信息，拿到用户信息中的所有路由权限码和本地预先声明的动态路由列表进行过滤筛选，把该用户有权限访问的路由过滤出来。然后动态生成路由记录。
  2. 后端直接返回用户所有有权限访问的路由配置信息，前端根据这些路由配置信息进行路由列表的动态生成路由记录。

页面路由权限一般搭配 路由导航守卫 来实现的。路由导航守卫有好几种，一般使用的是 全局前置守卫。

全局前置守卫在我们每次跳转路由页面之前都会触发，我们可以在 router.beforeEach() 里进行一系列的逻辑判断，从而判断该用户访问的路由页面是否能正常跳转。

路由全局前置守卫

我们可以在项目中的新建 src/router/permission.ts 文件，通过 路由全局前置守卫 来处理路由权限问题，当然这个 permission.ts 文件放在哪里都可以，我个人是习惯放在 src/router 目录下：

/**
 * 路由导航守卫模块
 */

import router from '@/router'
import { getToken } from '@/utils/auth'
import { useUserStore } from '@/store/user'
import usePermissionStore from '@/store/permission'

// 白名单路由
const whiteList = ['/login']

// 注意：不要在这里获取用户相关仓库
// 报错："getActivePinia()" was called but there was no active Pinia. Are you trying to use a store before calling "app.use(pinia)"?
// 因为在组件的外部，通过同步语句的方式获取仓库的数据是拿不到的，pinia 实例还未创建。
// 因为要获取小仓库的数据，必须要先有大仓库。
// const userStore = useUserStore()

// 可以通过以下方式获取，或者直接在 beforeEach 中获取
// import pinia from '@/stores'
// const userStore = useUserStore(pinia)

// 路由跳转前
router.beforeEach(async (to, from, next) => {
  const hasToken = getToken() // 获取用户登录后的 token
  
  // 如果有 token
  if (hasToken) {
    // 登录成功，如果跳转登录页，跳转到首页
    if (to.path === '/login') next({ path: '/' })

    // 如果跳转其他页面，判断用户的权限，是否有该页面路由的权限
    else {
  	  // 获取用户仓库相关信息
      const userStore = useUserStore()
      const hasRoles = userStore.roles && userStore.roles.length > 0

      // 如果该用户有角色信息
      if (hasRoles) {
        // 未匹配到任何路由，跳转404
        if (to.matched.length === 0) {
          // 未匹配到路由，并且跳转过来的原页面也找不到的话，跳转404，否则跳转回原来的页面。
          from.name ? next({ name: from.name }) : next('/404');
        } 
        else {
          console.log(`匹配到路由，放行：${to.path}`)  
          next()
        }
      }
	
      // 如果用户没有角色信息
      else {
        try {
          // 请求用户信息
          const data = await userStore.getUserInfo()
          // 获取 permission 的 store
          const permissionStore = usePermissionStore()
          // 调用 permission 的 store 中的生成该用户对应角色的权限路由方法
          const accessRoutes = await permissionStore.generateRoutes(data.roles)

          console.log('该用户的角色能访问的路由为：', accessRoutes)

          // 把生成的动态依次添加到路由中
          accessRoutes.forEach((route: any) => {
            // 添加一条新的路由记录到路由
            router.addRoute(route)
          })
          
          // 这里为什么要用 next({ ...to, replace: true })，可以看 注意事项 的说明
          next({ ...to, replace: true })
        } catch (error) {
          // 如果获取用户信息失败，清空用户登录的相关信息，并且重定向到登录页面
          console.log(error);
          await userStore.resetToken()
          next(`/login?redirect=${to.path}`)
        }
      }
    }
  }
  // 如果用户没有 token
  else {
    // 如果访问的路径在白名单中，直接放行
    if (whiteList.indexOf(to.path) !== -1) next()
    else next(`/login?redirect=${to.path}`)
  }
})

Pinia 仓库声明相关模块

用户信息模块

存放用户相关信息的仓库 src/store/user.ts：

import { defineStore } from 'pinia'
import { getToken, setToken, removeToken } from '../utils/auth'
import { login, getUserInfo } from '@/api/user'

// state 类型接口
interface IUserState {
  token: string,
  roles: string[]
}

interface UserInfo {
  userName: string,
  roles: string[]
}

interface ILogin {
  userName: string,
  password: string
}

export const useUserStore = defineStore('user', {
  // 声明 state，返回值类型是 IUserState
  state: (): IUserState => {
    return {
      token: getToken() || '',
      roles: [], // 用户角色列表
    }
  },
    
  actions: {
    setToken(token: string) {
      this.token = token
    },

    login(loginData: ILogin) {
      return new Promise<void>(async (resolve, reject) => {
        try {
          const res = await login(loginData)
          const { token } = res.data.token
          this.token = token
          setToken(res.data.token)
          resolve()
        } catch (error) {
          reject(error)
        }
      })
    },

    // 获取用户信息
    getUserInfo() {
      return new Promise<UserInfo>(async (resolve, reject) => {
        try {
          const res = await getUserInfo()
          this.roles = res.data.roles
          resolve(res.data)
        } catch (error) {
          console.log(error)
          reject(error)
        }
      })
    },

    resetToken() {
      this.token = ''
      this.roles = []
      removeToken()
    }
  }
})

权限信息模块

存放用户权限相关信息的仓库 src/store/permission.ts：

import { defineStore } from 'pinia'
import { RouteRecordRaw } from 'vue-router'
import { constantRoutes, asyncRoutes } from '@/router'
import { getRoutes } from '@/api/user'
const modules = import.meta.glob('../views/**/**.vue');
const Layout = () => import('@/layout/index.vue');

// 类型接口
interface PermissionState {
  routes: RouteRecordRaw[];
  addRoutes: RouteRecordRaw[];
}

/**
 * 递归过滤有权限的异步(动态)路由
 * @param routes 异步(动态)路由
 * @param roles 用户角色权限集合
 * @returns 返回用户有权限的异步(动态)路由
 */
const filterAsyncRoutes = (routes: RouteRecordRaw[], roles: string[]) => {
  // 存放过滤好的权限路由列表
  const res: RouteRecordRaw[] = [];

  routes.forEach(route => {
    const tmpRoute = { ...route }

    // 判断用户(角色)是否有该路由的访问权限
    if (hasPermission(roles, tmpRoute)) {
      // 1. 第一种情况的递归过滤（前端预先定义好动态路由的情况）
      // if (tmpRoute.children) tmpRoute.children = filterAsyncRoutes(tmpRoute.children, roles)

      // 2. 第二种情况（后端直接通过接口返回路由的情况）
      // 因为后端直接返回的是该用户有权限的路由，所以当使用第二种方式返回路由时，
      // 在当前这个方法中的作用不是为了筛选路由，而是给每个路由和子路由找到对应路径的组件。
      if (tmpRoute.component?.toString() === 'Layout') tmpRoute.component = Layout;
      else {
        const component = modules[`../views/${tmpRoute.component}.vue`];
        if (component) tmpRoute.component = component;
        else tmpRoute.component = modules[`../views/error-page/404.vue`]
      }

      // 如果有子路由，递归设置 components
      if (tmpRoute.children) tmpRoute.children = filterAsyncRoutes(tmpRoute.children, roles)
      
      // 将过滤后的路由添加到结果中
      res.push(tmpRoute)
    }
  })

  return res
}

/**
 * 通过路由中的 meta 路由元信息对象中的 roles 属性，进行判断用户是否有具有该路由权限
 * @param roles 用户角色集合
 * @param route 路由
 * @returns
 */
const hasPermission = (roles: string[], route: RouteRecordRaw) => {
  // 如果该路由有声明 路由元信息，并且有 roles 属性
  if (route.meta && route.meta.roles) {
    // 角色【超级管理员】拥有所有权限，忽略校验
    if (roles.includes('superAdmin')) {
      return true
    }

    // 如果不是超级管理员，通过用户的角色权限列表和路由中的权限进行匹对
    // 如果用户的角色权限列表中含有和该路由相同的权限，返回 true。
    return roles.some(role => {
      if (route.meta && route.meta.roles !== undefined) {
        return (route.meta.roles as string[]).includes(role)
      }
    })
  }

  return false
}

// 定义一个路由权限的 store，id 名称为 permission
const usePermissionStore = defineStore({
  id: 'permission',

  state: (): PermissionState => {
    return {
      addRoutes: [], // 动态路由
      routes: [] // 所有路由集合
    }
  },

  actions: {
    /**
     * 保存路由（静态路由 + 用户有权限的路由）
     * @param routes 用户有权限的路由列表
     */
    setRoutes(routes: RouteRecordRaw[]) {
      // 保存 用户的权限路由列表 到 动态路由 中
      this.addRoutes = routes

      // 合并静态路由和获取到的用户权限路由
      this.routes = constantRoutes.concat(routes)
    },

    /**
     * 生成动态路由
     * 有两种方式生成动态路由：
     * 1. 在前端预先定义好动态路由列表，然后进行匹配：
     *    在 router/index.ts 中定义动态路由时，需要给每个定义的路由的路由元对象 meta 中添加 roles 属性，
     *    roles 是一个字符串列表，列表中的角色编码表示有权限访问该路由的用户角色。
     *    接着请求后端接口，获取该用户的角色编码，然后在这里通过匹配 router/index.ts 中，
     *    预先定义的动态路由列表，如果列表中的路由的某一个角色编码和接口获取的用户某一个角色编码相同，
     *    添加到有权限访问的路由集合中。
     * 
     * 2. 前端不预先定义好动态路由列表，直接通过请求后端接口获取：
     *    通过请求接口获取该用户所有的权限路由列表，然后添加到有权限访问的路由集合中。
     * @param roles 用户角色集合
     * @returns 
     */
    generateRoutes(roles: string[]) {
      return new Promise<RouteRecordRaw[]>(async (resolve, reject) => {
        // 有权限访问的路由集合
        let accessedRoutes = <RouteRecordRaw[]>[]

        /*************************************************************************** */
        // // 1. 第一种方式：
        // // 如果用户的权限码含有 superAdmin，那么该用户有权限访问所有路由，
        // // 直接把所有前端定义的动态路由设置有有权限访问的路由集合。
        // if (roles.includes('superAdmin')) {
        //   accessedRoutes = asyncRoutes || []
        // }
        // // 如果用户的权限码不含有 admin，对每一个路由进行匹配
        // else {
        //   accessedRoutes = filterAsyncRoutes(asyncRoutes, roles)
        // }

        /*************************************************************************** */

        // 2. 第二种方式：
        // 直接通过请求后端接口返回该用户角色对应的动态路由
        const res = await getRoutes()
        accessedRoutes = filterAsyncRoutes(res.data.routes, roles)
        
        /*************************************************************************** */
        
        // 保存路由
        this.setRoutes(accessedRoutes)
        resolve(accessedRoutes)
      })
    }
  }
})

export default usePermissionStore

Token 的相关方法

src/utils/auth.ts：

export function setToken(token: string) {
  localStorage.setItem('token', token)
}

export function getToken() {
  return localStorage.getItem('token') || ''
}

export function removeToken() {
  localStorage.removeItem('token')
}

next() 说明

对于上述代码中，需要说明：

在路由守卫中，只有 next() 是放行，其他的诸如：next('/login') 、 next(to) 或者 next({ ...to, replace: true }) 都不是放行，而是：中断当前导航，执行新的导航。

中断就是此时不会执行 router.afterEach(() => {}) 这一次路由守卫的操作，而是又进入一次前置路由守卫，就像嵌套一样，一层路由守卫，然后又是一层路由守卫。

比方说：

如果我有一个守卫，在守卫中我使用 next('/login')，其实并不是直接跳转 /login，而是：

router.beforeEach(async (to, from, next) {
  router.beforeEach(('/login', from, next) {
    router.beforeEach(('/login', from, next) {
	  router.beforeEach(('/login', from, next)) {
        // .... 一直循环下去，因为我们没有使用 next()，而是使用 next('/login')
      }
	})
  })
})

所以每次在判断完 if 之后，必须要在 else 中进行放行，不然会造成死循环，如下：

if(to.path === '/login') next('/')
else next()

注意事项

这里有个注意事项，如果我们访问了某个动态路由，并且此时刷新页面，在 addRoute() 之后第一次访问被添加的路由会白屏。

这是因为 addRoute() 后就 立刻访问被添加的路由，此时 addRoute() 没有执行结束，因而找不到刚刚被添加的路由导致白屏。因此需要从新访问一次路由才行新访问一次路由才行。

此时就要用 next({ ...to, replace: true }) 来确保 addRoute() 时动态添加的路由已经被完全加载上去。

next({ ...to, replace: true }) 中的 replace: true 只是一个设置信息，告诉 VUE 本次操作后，不能通过浏览器后退按钮，返回前一个路由。

因此 next({ ...to, replace: true }) 可以写成 next({ ...to })。next({ ...to }) 的执行很简单，它会判断：如果参数 to 不能找到对应的路由的话，就再执行一次 beforeEach((to, from, next)，直到其中的 next({ ...to}) 能找到对应的路由为止。

因此需要用代码来判断这一次是否就是前往对应路由的 beforeEach((to, from, next)，如果是，就执行 next() 放行。如果守卫中没有正确的放行出口的话，会一直 next({ ...to}) 进入死循环！
