垃圾回收 和 内存泄露 和 闭包

垃圾回收

要聊清楚 垃圾回收，首先得知道 什么叫垃圾，这个概念一直以来没有一个明确的定义，不过形成的共识就是：不再需要的内存，这里内存里面的数据就是垃圾。

那么，什么又是不再需要呢？这个不再需要是由谁来决定的呢？其实由人决定的？我们说它需要就需要，说它不需要就不需要。

所以，这件事就有的时候看着比较明确，有的时候就没那么明确。因为认为的因素太多了。

比方说一下例子：

function createIncrease() {
  const doms = new Array(1000).fill(0).map((item, i) => {
    const dom = document.createElement('div')
    dom.innerHTML = i
    return dom
  })

  function increase() {
    doms.forEach(dom => {
      dom.innerHTML = Number(dom.innerHTML) + i
    })
  }
    
  return increase
}

const increase = createIncrease()
const btn = document.querySelector('button')
btn.addEventListener('click', increase)

上述代码是一个典型的闭包例子。现在整段的这个代码里边，这些 dom 元素是不是垃圾？答案肯定是否定的。

因为这些 dom 元素是我要用到的，我还需要它们。每次点击按钮的时候，都要调用 increase 函数，而这个 increase 函数又需要用到这些 dom 元素。

垃圾不是看它占用内存空间的大小，跟大小没关系，而是看你还需不需要。这个例子还是比较容易看出来的，因为有点击事件。

再来看另一个例子：

const nums = [1, 2, 3, 4, 5]
const sum = nums.reduce((total, num) => total + num, 0)
console.log(sum);

观察上诉代码，当代码运行完成后，nums 这块存内存我们是需要呢，还是不需要？这里其实是看不出来的，因为这里就三行代码，有可能后边还会有代码中会用到这里的 nums 或者 sum，也有可能不需要。

可能有人会说，就这里的三行代码而言，这里存储 nums 和 sum 变量的内存都不再需要了，后边已经没有代码了。如果我们把这段代码运行起来，浏览器打印一个 sum 的结果后，代码运行完成了。

然后，如果我们在控制台中通过 nums 直接访问该变量，控制台还是会打印 nums 变量存储的内容。那么，如果你说这个内存不需要了，但是现在却把 nums 内存里保存的东西打印出来了，也就是说我用到了这个 nums 了，即这个 nums 内存还是有用的。

所以，这个需不需要，有的时候在代码里可以看出来，有时候看不出来。

因此，是不是垃圾这件事情，得问你自己，是你在写代码，你是非常清楚这些东西以后还要不要用到。

于是，在内存里边就会形成这么一种状态。

比方说上图中，是我们整个程序运行过程中的内存，内存里边的很多东西我们都是需要的，但是有一些东西我们可能不再需要了。

然后我们再说 垃圾回收，因为在 js 中是有一个 垃圾回收器，它可以帮助我们回收那些我们不再需要的内存。那么问题又来了，有时候我们在写代码时自己都不知道还需不需要某些内存，它怎么知道我们不再需要了？

其实它根本就不知道你是否还需要这些内存，比方说第一个例子中，它敢回收那些点击事件用到的内存存储的变量吗？它如果回收了，将来点击按钮的时候就用不了那些 dom 元素了。再说第二个例子中，它如果回收了，我在控制台中访问某个变量时候，不就没办法显示对应内容了？

既然它不知道我们是否需要这些内存？那它怎么办呢？其实，它做了一个假设，虽然它不知道我们需要那些内存，不需要哪些内存，但是 它可以确定一点，就是有一些东西你一定是不想要的。

比方说第二个例子中：

const nums = [1, 2, 3, 4, 5]
nums = [6, 7, 8, 9] // 重新赋值
const sum = nums.reduce((total, num) => total + num, 0)
console.log(sum);

上诉代码中，我们对之前的代码做了修改，在第二行中，我们对 nums 进行了重新赋值，那么第一行中数组的内存空间，我们还有机会去访问到它吗？已经是永远无法访问到它了，那么 垃圾回收器 就会认为这些东西你一定是不要的了。

所以，垃圾回收器 能回收哪些东西？它回收的是哪些我们无法触达的内存空间。相当于这些内存空间是我们整个不想要的内存空间里的一个子集。这就是 垃圾回收器 要干的事。

内存泄露

什么叫内存泄露呢？垃圾回收器 把那些我们无法触达的内存回收掉了，但是在整个运行过程中的内存中，还是会剩下一些我们不需要的内存，只是说这些我们不需要的内存我们目前还是可以触达的，如下图所示：

上图中，还剩下的我们不需要的、但是还能触达的内存就是 内存泄露。当这个 内存泄露 变大以后就会严重影响我们程序的运行。

所以说，在 JS 里边，如果来解决 内存泄露？就是让这些内存重新变得不可触达。因为它一旦不可触达了，那么 垃圾回收器 就会发现并把它回收掉。

比方说之前的例子中：

const nums = [1, 2, 3, 4, 5]
const sum = nums.reduce((total, num) => total + num, 0)
console.log(sum);

如果说我们已经确认不再需要 nums 内存中存储的 [1, 2, 3, 4, 5] 了，那么可以把 nums 赋值为 null。那么，我们就永远无法触达到之前的 [1, 2, 3, 4, 5] 了，也就会被 垃圾回收器 给回收掉。

onst nums = [1, 2, 3, 4, 5]
const sum = nums.reduce((total, num) => total + num, 0)
console.log(sum);
nums = null

在比方说最开始的例子：

function createIncrease() {
  const doms = new Array(1000).fill(0).map((item, i) => {
    const dom = document.createElement('div')
    dom.innerHTML = i
    return dom
  })

  function increase() {
    doms.forEach(dom => {
      dom.innerHTML = Number(dom.innerHTML) + i
    })
  }
    
  return increase
}

const increase = createIncrease()
const btn = document.querySelector('button')
btn.addEventListener('click', increase)

如果说我们明确是不再需要变量 increase 和 btn 的点击事件了，可以做以下处理：

function createIncrease() {
  const doms = new Array(1000).fill(0).map((item, i) => {
    const dom = document.createElement('div')
    dom.innerHTML = i
    return dom
  })

  function increase() {
    doms.forEach(dom => {
      dom.innerHTML = Number(dom.innerHTML) + i
    })
  }
    
  return increase
}

const increase = createIncrease()
const btn = document.querySelector('button')

function handleClick() {
  increase()
  
  // 移除事件的监听
  btn.removeEventListener('click', handleClick)
  
  // 同时把 increase 设置为 null
  increase = null
}

btn.addEventListener('click', handleClick)

上诉代码，一旦把 increase 设置为 null ，那么我们就永远无法触达到之前的 increase 函数的内容，那么之前 increase 函数内涉及到的 doms 也会被 垃圾回收器 回收掉。

闭包的内存泄露

闭包和内存泄露有什么关系呢？从目前看来好像没什么关系，只是闭包容易让我们放松警惕，比方说之前的例子中的 nums，如果不需要了就可以直接设置为 null。

但是闭包需要绕一下，比方说变量 increase 是一个函数，将来我们可能不再用这个函数了，但是我觉得函数占不了多少内存空间，所以就容易掉以轻心，在不再需要它的时候，没有把它的引用设置为 null，也就是没有 increase = null。我们可能会觉得函数相对于数组和对象等引用数据而言，占用内存小，漏了设置就漏了吧。

但是，这一漏这个函数背后的那个闭包环境中，它关联的这个 doms 也保留下来了。因为你还可以通过 increase 函数间接地触达到这个 doms。于是就导致了 内存泄漏。

闭包还有一个非常隐蔽的泄漏点。我们之前觉得垃圾回收机制很智能，一旦这个内存空间无法触达了，它就会自动帮我们回收掉。那么有没有一种情况就是那个内存空间无法触达，但是它仍然无法回收掉呢？它是有的。这种情况有可能是来源一些浏览器的 BUG，还有一种情况就是来源于闭包。

这种情况是什么情况呢？我们在之前的例子上做一些改造：

function createIncrease() {
  const doms = new Array(1000).fill(0).map((item, i) => {
    const dom = document.createElement('div')
    dom.innerHTML = i
    return dom
  })

  function increase() {}
  
  function _temp() {
      console.log(doms)
  }
    
  return increase
}

let increase;
const btn = document.querySelector('button')

btn.addEventListener('click', () => {
  increase = createIncrease()
})

在上诉代码中，我们对 createIncrease 函数内部进行了修改，内部的 increase 函数不再引用 doms，并且添加一个新的函数 _temp，_temp 内部引用了 doms，但是只返回了 increase 函数。

在点击按钮的时候，对变量 increase 进行赋值，那么代码在后续运行期间，我们还有没有可能触达到这个 doms？是绝无可能触达到的。因为 _temp 并没返回给外界使用，increase 又没有引用 doms。

所以这里的 doms 是一块无法触达的内存空间，那么这块内存空间能够被回收吗？其实是不能的。为什么呢？

是因为 _temp 函数造成的，_temp 和 increase 的闭包环境是一样的，它们都共享同一个 词法环境，如果说没有 _temp，只有这个 increase 的话，那么浏览器会做优化，这个 词法环境 里面有什么呀？有一个 doms，但是这个 doms 没有被 increase 用到，没被用到的话，那么就把 doms 从 词法环境 中优化掉，那么这个 词法环境 就会变成空的，里面什么也没有，这样确实不会造成内存泄露。

但是，如果说多了一个 _temp，并且还引用了 doms，那么浏览器就不敢把 doms 从这个 词法环境 中去掉。

createIncrease 函数中只返回了 increase，我们访问不到 _temp，_temp 就被回收掉了，但是这个 词法环境 中仍然保留了这个 doms。因为到了这个时间点，浏览器之前的优化词法环境的步骤已经结束了，它不会重复地去优化它了。

从而就导致了这个 increase 里边尽管没有用到这个 doms，但是受另一个函数的影响，doms 仍然存在这个 此法环境 里边。从而导致了一个 闭包非常隐蔽的内存泄露。

闭包会导致内存泄露的关键点：

  1. 闭包和其它的内存泄露没有其他本质区别。都是因为持有了不再需要的函数引用，会导致函数关联的词法环境无法销毁。从而导致内存泄漏。
  2. 当多个函数共享同个 词法环境 时，会导致词法环境膨胀，从而导致出现无法触达也无法被回收的内存空间。

最后，我们工作当中的心智负担是不是有点太大了。因为在我们平时开发当中，闭包是必定会遇到的。但是要记住一点：永远不要提前优化。永远都是出了问题后再去调试，看哪一块内存占用过高，该销毁的没销毁。
