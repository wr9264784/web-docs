闭包（Closure）

闭包有哪些容易引起歧义的地方？

  • 闭包一定有 return 吗？（答案是不一定）
  • 闭包一定会内存泄露吗？（答案是不一定）

闭包的概念

概念：闭包（closure）是一个函数以及其捆绑的周边环境状态（lexical environment，词法环境）的引用的组合。换而言之，闭包让开发者可以从内部函数访问外部函数的作用域。在 JavaScript 中，闭包会随着函数的创建而被同时创建。

简单理解：闭包 = 内层函数 + 引用的外层函数变量。

先看个简单的代码：

function outer() {
  const a = 1
  function f() {
    console.log(a);
  }
  f()
}

outer()

说明上述的代码中存在 闭包。形成闭包作用域的名称为外部的 outer 函数提供的作用域，上述的代码也是最简单中闭包形式。

通常会再使用一个函数包裹住闭包结构，以起到对变量保护的作用。

闭包什么时候用到 return

外部如果想要使用闭包的变量，则此时需要 return

function outer() {
  const a = 10 // 闭包内的变量
  return function () {
    console.log(a);
  }
}

const fn = outer()
fn() // 10

先来看一个例子：

let i = 0
function fn() {
  i++
  console.log(`函数被调用了${i}次`);
}

fn() // 函数被调用了1次
fn() // 函数被调用了2次

i = 1000
fn() // 函数被调用了1001次

上述例子中的代码是没有问题的，但是 i 是一个全局变量，如果此时对 i 进行了修改，然后再调用 fn，那么打印的结果就不准确。

为解决上述的缺陷，我们可以采用 闭包 的形式，因为 闭包最典型的一个应用就是可以实现数据的私有化，比如，我们要做一个统计函数调用次数的方法，函数调用一次，内部变量就 ++：

function count() {
  let i = 0
  return function fn() {
    i++
    console.log(`函数被调用了${i}次`);
  }
}

const fun = count()

fun() // 函数被调用了1次
fun() // 函数被调用了2次

i = 1000
fun() // 函数被调用了3次

闭包的内存泄露

谁会存在内存泄露？

function fn() {
  let count = 1
  return function fun() {
    count++
    console.log(`函数被调用了${count}次`);
  }
}

const result = fn()

result() // 函数被调用了2次
result() // 函数被调用了3次

借助于垃圾回收机制的 标记清除法 可以看出：

  1. result 是一个 全局变量，代码执行完毕不会立即销毁。
  2. result 使用了 fn 函数。
  3. fn 用了 fun 函数。
  4. fun 函数用到了 count 变量。
  5. count 被引用就不会被回收，所以一直存在。

此时，闭包引起了 内存泄露。

那么应该怎么解决 内存泄露 呢？

我们可以 在 fn 函数执行完毕的时候手动释放，即 将 result 变量置为 null，这样 赋值给 result 的 fn 函数就没有被其他变量引用了，就会被回收了。

注意：

  1. 不是所有的内存泄露都要手动回收的。
  2. 比如 react 里面很多闭包都不能回收的。
