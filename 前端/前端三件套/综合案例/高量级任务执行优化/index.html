同步执行

如果我们直接执行（也就是同步执行） 1000 个耗时任务，那么 index.js 如下：

let isRun = false // 判断是否已经点击了按钮
let num = 0 // 已执行的任务数量
let isPromise = false // 判断是否是异步任务

/**
 * 运行一个耗时任务
 * 1. 如果要异步执行任务，请返回 Promise，
 * 2. 要尽快完成任务，同时不要让页面产生卡顿。
 * 3. 尽量兼容更多的浏览器
 * @param { Function } task 执行的任务
 */
function runTask(task) {
  // 1. 直接运行任务，会阻塞页面
  // 阻塞原因很简单，因为这是一个同步任务，由于执行了 1000 次，比较耗时，
  // 所以就导致页面阻塞。
  task()
}

const btn = document.querySelector('button')
const finished = document.querySelector('.finished')

btn.onclick = async () => {
  // 如果已经点击了按钮执行任务，直接退出。避免造成多次任务重复执行
  if (isRun) {
    alert('已经在执行啦。')
    return
  }

  const start = Date.now()

  // 调用 1000 次耗时任务
  for (let i = 0; i < 1000; i++) {
    const res = runTask(task)

    // 如果返回结果有 then 方法，说明是 Promise 对象
    if (res && res.then) {
      isPromise = true
      res.then(() => {
        const end = Date.now()
        num++

        // 如果任务是异步的，那么就等待所有任务执行完成后再计算完成时间
        if (num >= 1000) {
          finished.innerHTML = `执行完成，耗时${end - start}毫秒`
        }
      })
    }
  }

  // 如果任务不是异步的，那么直接计算完成时间
  if (!isPromise) {
    const end = Date.now()
    finished.innerHTML = `执行完成，耗时${end - start}毫秒`
  }

  isRun = true
}

/**
 * 模拟一个耗时任务
 */
function task() {
  const start = Date.now()
  while(Date.now() - start < 5) {}
}

从上图中，我们可以看到当我们点击按钮后，小球会停止运动，一段时间后小球恢复执行，并且输出了任务执行完成的时间。

同步执行会造成浏览器的阻塞，这是非常好理解的，因为这里是同步代码，调用 1000 次就是执行了 1000 次 runTask 方法，也就意味着 runTask 的执行会阻塞页面的后续渲染。因为 js 是 单线程 的。

异步执行 - 微任务

根据上述的同步执行，那么我们会想：能不能把它变成异步呢？异步执行的话 runTask 方法要返回一个 Promise，目的主要是为了统计执行时间。

异步执行，我们修改 index.js 中的 runTask：

/**
 * 运行一个耗时任务
 * 1. 如果要异步执行任务，请返回 Promise，
 * 2. 要尽快完成任务，同时不要让页面产生卡顿。
 * 3. 尽量兼容更多的浏览器
 * @param { Function } task 执行的任务
 */
function runTask(task) {
  // 2. 异步的形式，把这个任务放到异步队列里执行
  return new Promise((resolve, reject) => {
    Promise.resolve().then(() => {
      // 在微队列中执行任务
      task()
      resolve()
    })
  })
}

发现在 微任务队列 中执行，页面仍然会阻塞。

为什么微任务也会阻塞呢？这就涉及到 事件循环 了，我们得知道事件循环是如何来处理 微任务 的？就是遇到 微任务 就一定要 把微任务这个队列全部清空，才能做后面的事，包括页面渲染。那么这就会影响到 渲染帧。

渲染帧 是 16ms 一帧，如果在某一帧里边出现了大量的微任务，那么就会导致这个 渲染帧 延后，它一定要把这个 微任务 全部执行完成了，才会去进行渲染。

所以，微任务 也会造成阻塞。

异步执行 - setTimeout

既然 微任务 不行，那么我们使用定时器可不可以呢？

我们先来修改一下 index.js 中的 runTask：

/**
 * 运行一个耗时任务
 * 1. 如果要异步执行任务，请返回 Promise，
 * 2. 要尽快完成任务，同时不要让页面产生卡顿。
 * 3. 尽量兼容更多的浏览器
 * @param { Function } task 执行的任务
 */
function runTask(task) {
  // 3. 异步的形式，把这个任务放到 setTimeout 里面执行，
  // 发现使用 setTimeout 执行任务后，页面没有阻塞，但是会有卡顿。
  return new Promise((resolve, reject) => {
    setTimeout(() => {
      task()
      resolve()
    }, 0)
  })
}

页面效果如下所示：

使用 setTimeout 后发现没有造成阻塞，但是页面会有严重的卡顿现象。

setTimeout 按照过去的说法会产生 宏任务，这个任务会进入 宏队列。按照现在的浏览器的 事件循环 的最新说法，已经 不再使用宏任务 这种说法了，而是改用更加精细的任务队列 - 延时任务，这个任务会进入 延时队列。

那么 延时任务 为什么会造成卡顿呢？那么首先就要知道 延时任务 是怎么执行的。

事件循环 实际上就是一个死循环，我们单说 延时任务 ，它会在这个循环里做什么事呢？它会取出 延时任务 中的第一个任务，就是最先加入这个队列的任务。然后执行任务。再然后接下来判断是否到达渲染时机，如果渲染时机到达了就渲染。

可以简单的理解为到了 16.6ms 它就要进行渲染了。所以，为什么 延时任务 不会造成页面阻塞，是因为它每次只取一个任务执行，执行完了之后，如果渲染时机到了就渲染，然后下一次循环再取下一个任务。所以它就不会造成页面阻塞。

那么它为什么会卡顿呢？那是因为不同的浏览器判断这个渲染的时机的方式不同，W3C官网 并没有明确的说这个渲染时机，它大概说了一下。很多时候需要浏览器自己去判断。然而每个浏览器的判断方式又有差异。

像谷歌浏览器，它会认为现在这个队列里面有这么多的任务，有这么高的计算需求。它觉得需要匀一部分计算资源给这些正在等待的 延时任务 。所以，它会稍微的把这个渲染时机往后延迟一下。导致我们看到的页面会有卡顿。

对于 edge 浏览器的处理方式和谷歌差不多。谷歌浏览器的这种做法也是说的过去的，不过在有些例子里就不合适了，像当前的这个案例中，没有什么需要渲染的任务，就一个小球的渲染动画。

所以呢在这个案例中，浏览器把渲染时机延迟就不合适了。但是，如果说在一个非常庞大的网页里面，渲染任务很重，那么它交这个渲染时间，对于尽快计算完所有的任务是有帮助的。

但是注意，对于 safari 浏览器，它是不会阻塞渲染，也不会造成卡顿的。也就是说它在判断渲染时机是否达到的时候，不考虑有多少个任务需要执行。它该渲染就渲染，所以它的耗时也就是任务完成的时间会多一点。

异步执行 - requestAnimationFrame

既然 延时任务 不行，那么能不能用 requestAnimationFrame 来实现呢？我们可以先来试一下，修改 index.js 中的 runTask：

/**
 * 运行一个耗时任务
 * 1. 如果要异步执行任务，请返回 Promise，
 * 2. 要尽快完成任务，同时不要让页面产生卡顿。
 * 3. 尽量兼容更多的浏览器
 * @param { Function } task 执行的任务
 */
function runTask(task) {
  // 3. 异步的形式，把这个任务放到 requestAnimationFrame 动画帧里执行,
  // 发现页面仍然会造成阻塞.
  return new Promise((resolve, reject) => {
    requestAnimationFrame(() => {
      task()
      resolve()
    })
  })
}

经过测试，发现页面同样会出现阻塞，为什么会阻塞呢？因为在渲染帧之前执行了 1000 次这个耗时任务，那就无限地把这个渲染帧往后延迟了。

比如说页面在某一帧本来是要渲染的，但是在这个位置执行了 1000 次耗时任务。那这个渲染任务肯定是要被延后的。必须要等待任务执行完成后才能渲染，这是由于 JS 单线程导致。

异步执行 - 手动控制执行时机

这个时候我们就要这样来考虑了，这个任务要执行还是不执行取决于目前这一帧还剩余多少时间。一帧是 16.6ms，只要还剩余有时间那么就执行任务。如果不剩余时间那么就不执行，推迟到下一帧执行。

因此，为了完成这件事，我们可以在 index.js 中写一个辅助函数叫做 _runTask：

/**
 * 写一个辅助函数，手动控制任务的执行时机，
 * @param { Function } task 要执行的任务
 * @param { Function } callback 如果任务执行了，需要传一个回调
 * 
 * 如果将来 task 执行了，_runTask 函数会执行 callback。
 */
function _runTask(task, callback) {
  // ...后续补充
}

然后再修改 index.js 中的 runTask 方法，在 runTask 中调用 _runTask：

/**
 * 运行一个耗时任务
 * 1. 如果要异步执行任务，请返回 Promise，
 * 2. 要尽快完成任务，同时不要让页面产生卡顿。
 * 3. 尽量兼容更多的浏览器
 * @param { Function } task 执行的任务
 */
function runTask(task) {
  // 4. 异步的形式，使用辅助函数帮助我们判断任务的执行时机
  return new Promise((resolve) => {
    // 把任务和 resolve 传入进去，一旦 task 被调用了 _runTask 方法会帮我们调用 resolve
    _runTask(task, resolve)
  })
}

接下来要继续补充 _runTask 方法，在 _runTask 中我们需要先判断一下是否在某一帧中还有剩余时间，如果有就执行任务以及对应传入的回调，如果没有，就推迟到下一帧执行。

那么问题是，我们如何知道当前帧是否还有剩余时间呢？这就要用到 requestIdleCallback 了，它跟 requestAnimationFrame 的含义差不多，都是每一帧渲染的时候有一个回调，但是通过它可以拿到当前渲染帧还剩余多少时间，例如：

requestIdleCallback((idle) => {
  console.log(idle);
})

打印结果如下：

我们通过 [[Prototype]] 上的 timeRemaining 方法就可以获取到剩余时间：

requestIdleCallback((idle) => {
  console.log(idle.timeRemaining()); // 15.9
})

这个时间如果大于 0，说明一个渲染帧还有剩余时间。所以，我们就可以通过这个时间的大小来判断是否还有剩余时间。

好了，有了以上的知识，我们来修改 index.js 中的 _runTask：

/**
 * 写一个辅助函数，手动控制任务的执行时机，
 * @param { Function } task 要执行的任务
 * @param { Function } callback 如果任务执行了，需要传一个回调
 * 
 * 如果将来 task 执行了，_runTask 函数会执行 callback。
 */
function _runTask(task, callback) {
  requestIdleCallback((idle) => {
    // 首先要判断一下当前帧是否还有剩余时间
    if (idle.timeRemaining() > 0) {
      // 如果有剩余时间，就执行任务以及对应的回调
      task()
      callback()
    }

    // 如果没有剩余时间，就推迟到下一帧再执行
    else {
      // 如果没有剩余时间，递归调用自己，继续判断下一帧是否有剩余时间
      _runTask(task, callback)
    }
  })
}

  // 如果已经点击了按钮执行任务，直接退出。避免造成多次任务重复执行
  if (isRun) {
    alert('已经在执行啦。')
    return
  }

  const start = Date.now()

  // 调用 1000 次耗时任务
  for (let i = 0; i < 1000; i++) {
    const res = runTask(task)

    // 如果返回结果有 then 方法，说明是 Promise 对象
    if (res && res.then) {
      isPromise = true
      res.then(() => {
        const end = Date.now()
        num++

        // 如果任务是异步的，那么就等待所有任务执行完成后再计算完成时间
        if (num >= 1000) {
          finished.innerHTML = `执行完成，耗时${end - start}毫秒`
        }
      })
    }
  }

  // 如果任务不是异步的，那么直接计算完成时间
  if (!isPromise) {
    const end = Date.now()
    finished.innerHTML = `执行完成，耗时${end - start}毫秒`
  }

  isRun = true
}

/**
 * 模拟一个耗时任务
 */
function task() {
  const start = Date.now()
  while (Date.now() - start < 5) { }
}
