JWT

我们要认识一个东西，都先要知道这个东西到底解决了什么样的问题，它为什么会出现。同样的道理，我们要认识 JWT 就要先知道没有 JWT 的时候到底有什么样的问题。

一、没有 JWT 的情况

我们先看下图：

有的时候，服务器会发送一些信息给浏览器，比方说登录成功之后有些身份信息会让浏览器去保存。给了浏览器之后，浏览器就会把这个信息保存下来，具体的保存位置可以是 cookie、可以是 localStorage、可以是任何东西。到了接下来某一个时刻，如果浏览器要请求服务器的时候，它会把这个保存的信息读取出来，然后通过 http 协议发送给服务器。服务器就会知道了，浏览器有这个身份信息，于是表明它是有登录过的。

在这种简单的模式下，会有一个什么问题呢？由于这个身份相关的信息是保存在浏览器端的，那么如果浏览器要去修改这个东西实在太简单了，包括 cookie、storage、indexDB 等都可以进行修改。所以，在这种模式下，就很难防止数据被篡改。这个更改后的信息传给服务器，服务器是不知道有没有被更改过的。

除了被篡改外，这个用户信息还可以被伪造，用户根本就没登录，但是通过浏览器或一些工具直接给它添加一些信息，只要我知道服务器的规则，就可以很轻松地伪造身份信息。然后发送请求的时候就把这个身份信息携带过去了。

所以，无论是篡改还是伪造，它都表明了一件事：就是 服务器无法信息这个信息。服务器无法判断和验证这个信息 是一个正常的信息，还是一个被篡改过或伪造过的信息。

二、签名

在服务器无法信任客户端传递过来的信息的情况下，我们要怎么办呢？我们可以想到一个办法，就是 签名。我们可以先看以下图例：

首先，签名是一种算法，你只要给它一个信息和一个密钥，然后通过某一种算法，就可以得到一个签名结果。通常我们会把这个结果转换成字符串来表示。

这个算法其实有很多，比较常见的是 HS256，我们可以通过以下例子说明进一步了解签名：

const crypto = require('crypto')

/**
 * 签名函数
 * 传入一些信息和密钥，生成一个签名。
 * @param { any } info 产生签名的信息
 * @param { any } key 密钥
 */
function sign(info, key) {
  const hmac = crypto.createHmac('sha256', key)
  hmac.update(info)
  return hmac.digest('hex')
}

// 测试生成密钥，密钥一般是服务器端生成，保存在服务器的
const res = sign('测试签名', '123456')
console.log(res); // 结果为；105e4045c0781af485ed67cacd6a03bf571891e075c9aa5e5fefcb6f5dade99f

理解了 签名 之后，再回过头来看上图：服务器要发送这个身份信息之前，首先会对这个身份信息进行签名，拿到签名的结果。同时，它会把这个 签名 和 身份信息 一起发送给浏览器。那么，浏览器就会拿到两个东西，一个是 身份信息、一个是签名，浏览器保存的时候都一起保存好。

保存好之后，到了下一次要请求服务器的时候，浏览器要把之前保存的 身份信息 加上 签名 一起发送给服务器。这个时候就和之前简单地把用户信息发给服务器不一样了，服务器通过这个信息本身就能验证信息有没有被篡改。

服务器是怎么验证的呢？在这个发送请求里包含两个部分，一个是 签名，一个是 身份信息。它只需要把这个 身份信息 用同样的密钥再来一次签名，然后把这个签名结果和浏览器发送过来的签名进行匹对，两个一样就验证通过，否则验证失败，这个请求的用户信息就不被信任。

如果说用户想直接在客户端对这个用户信息手动进行一次签名，然后再发送给服务器，那也是不行的，因为客户端就算进行了签名，但是客户端签名时用到的密钥和服务端使用的 密钥不一样，生成的签名结果就不可能相同。

那么从此以后，客户端传递的这一块东西（info 和 签名）对于服务器来说就是可被验证的。这种逻辑其实就是 JWT 的雏形。

三、JWT

jwt 出来之后，不需要手动去设置这些签名了。jwt 给我们定了一套标准：这一块应该怎么做、应该用什么样的签名算法、签名之后这个整个信息该如何来组织等等。但它的基本逻辑就是那一套签名的逻辑。

jwt 的本质其实就是一个非常长的字符串，如下图所示：

这个字符串由 3 个部分组成，每个部分用 . 进行分割。在实际情况下它是没有换行，图里的换行只是为了能看的清楚而已。这 3 个部分分别是：header（头）、payload（负荷）、signature（签名）。那么，这三部分到底是一个什么东西呢？有了前边的理论后，再看这一块就非常的简单了。

  • header 就是一个 json 字符串，它里边包含了整个 JWT 里边的签名算法，这里 alg 通常写的是 HS256，然后指明一个类型 type 叫做 JWT，基本上是一个固定的写法，就是一个 json 字符串。然后使用 base64 编码，就得到了 header。但是要注意：这里的 base64 不是加密，它只是一种格式转换，它可以很轻松地被解码还原出来。
  • 例如：
const str = '{ "alg": "HS256", "type": "JWT" }'

// base64 编码
const enStr = btoa(str)
console.log(enStr); // 'eyAiYWxnIjogIkhTMjU2IiwgInR5cGUiOiAiSldUIiB9'

// 解码
const deStr = atob(enStr)
console.log(deStr) // '{ "alg": "HS256", "type": "JWT" }'

  • payload 就是客户端想要传递给服务器的信息了，比方说用户的身份信息。这些信息也是一样需要转换成 json 格式，再经过 base64 编码就可以得到这个 payload。其实这个 payload 可以传递很多东西，比方说整个 jwt 的过期时间等。

  • signature 就是签名。它是怎么签名的呢？它是把这个经过 base64 编码的 header 和 payload 之间通过一个 . 进行连接，形成一个完整的字符串，然后再用这个 完整的字符串 加上 一个 密钥生成的签名结果。就像我们的在第二大点中描述的签名过程一样。最后还是一样要把这个签名的结果转换成 base64 编码后得到了这个 signature。

所以，jwt 本质就是一个很长的字符串。只是这个字符串是一个标准格式，有三个部分组成。我们理解这个原理之后就可非常轻松地自己来写一个 JWT 函数。

const crypto = require('crypto')

/**
 * 签名函数
 * 传入一些信息和密钥，生成一个签名。
 * @param { any } info 产生签名的信息
 * @param { any } key 密钥
 */
function sign(info, key) {
  const hmac = crypto.createHmac('sha256', key)
  hmac.update(info)
  return hmac.digest('hex')
}

/**
 * 手写一个 jwt 函数
 * @param { any } info 产生签名的信息
 * @param { any } key 密钥
 */
function jwt(info, key) {
  const header = {
    type: 'JWT',
    alg: 'HS256'
  }

  const headerStr = Buffer.from(JSON.stringify(header)).toString('base64')
  const payloadStr = Buffer.from(JSON.stringify(info)).toString('base64')
  const signRes = sign(headerStr + '.' + payloadStr, key) // 得到签名
  const signStr = Buffer.from(signRes).toString('base64')
  return headerStr + '.' + payloadStr + '.' + signStr
}

最后，按照 JWT 的原理，你会发现这里边好像并没有加密，第一部分和 第二部分 是明文，base64 编码可以很轻松解码出来。第三部分是一个签名，签名和加密还是有区别的，这个签名是用来验证前面两个部分有没有被篡改或伪造的。
